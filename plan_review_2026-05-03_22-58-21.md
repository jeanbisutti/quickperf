## Summary

The plan is the v2 iteration after a prior review and most of the structural corrections from that review have been folded back in (`ProxyFactoryFactory` for column count, opaque connection ids in `ConnectionLeakListener`, constructor-time registration in `TestConnectionProfiler`, per-`QueryInfo` `long[]` for column count, `@Execution(SAME_THREAD)` for R2DBC ITs, `QuickPerfR2dbcProxyMarker` preservation in the CGLIB path). The strategic shape is now sound and the `r2dbc-proxy` extension hook is the right one to use in principle.

But three concrete blockers remain that prevent any of PR-0/PR-2/PR-4 from compiling or running as written:

1. **`io.r2dbc.proxy.callback.ResultCallbackHandler` is `public final class`** — the plan's central design at §2.1 line 81 ("`QuickPerfResultCallbackHandler` that **extends** `io.r2dbc.proxy.callback.ResultCallbackHandler`") will not compile.
2. **`ConnectionListenerRegistry` reads from `InheritableThreadLocal`** — the §2.3 dispatch loop on Reactor scheduler threads will see an empty (or stale) listener set, so `ConnectionLeakListener` and `TestConnectionProfiler` will receive zero R2DBC events at runtime regardless of `@Execution(SAME_THREAD)`. The plan's deferred-to-v3 `Reactor Context` solution is the real fix; until then the design does not work.
3. **`ProxyFactory` is a 7-method interface with no public default implementation** — `JdkProxyFactory` and `JdkProxyFactoryFactory` are package-private final, so a custom `ProxyFactoryFactory` cannot delegate to the default. The plan's "wraps every `Result` produced" statement glosses over the fact that we have to implement (or copy) all 7 wrap methods.

Beyond these, the plan still cites several APIs that don't exist as stated (`createResultProxy` instead of `wrapResult`, `Result.getRowMetadata()`, `ConnectionPool extends AbstractConnectionFactory`), and the wrong package for `SqlAnnotationsConfigs`. The `SystemProperties` Spring-resolution claim in §3.5 is also wrong — those properties only resolve via `System.getProperty()`. Fix the three blockers, the API names, and the package paths, and the plan is executable.

## Blocker findings (must be addressed before PR-0 starts)

### B1 — `ResultCallbackHandler` is `final`; the chosen design does not compile

**Plan claim (§2.1.A, line 81):**
> "In `sql-annotations-r2dbc`, define `QuickPerfResultCallbackHandler` that extends `io.r2dbc.proxy.callback.ResultCallbackHandler` and intercepts `map(BiFunction<Row, RowMetadata, T>)` and `getRowsUpdated()`."

**Evidence:** `target/r2dbc-proxy-sources/io/r2dbc/proxy/callback/ResultCallbackHandler.java:40`:

```java
public final class ResultCallbackHandler extends CallbackHandlerSupport {
```

The class is `final`. `extends ResultCallbackHandler` cannot compile. Every subclass-based design point downstream (the §2.1 "Files" inventory entry `QuickPerfResultCallbackHandler`, the §5 unit test `QuickPerfResultCallbackHandlerTest`, the §6.1 R-rows that mention "the handler") inherits this defect.

**Why it matters:** PR-2 cannot start. There is no plausible fix at the `ResultCallbackHandler` level — the only supported seam is the higher-level `ProxyFactory.wrapResult` (see B3), where you write your own `Result` decorator class (a regular `final class implements io.r2dbc.spi.Result`) that delegates to the underlying `Result` and instruments `map(...)`/`flatMap(...)`. That decorator does **not** extend `ResultCallbackHandler`.

**Fix:**
- Rename the new class to `QuickPerfMonitoringResult` (or similar) and make it `final class QuickPerfMonitoringResult implements io.r2dbc.spi.Result`.
- Document that we do **not** extend `ResultCallbackHandler`; we write a `Result` adapter in our own package, returned from a custom `ProxyFactory.wrapResult(...)`.
- Update §2.1.A, the "Files" inventory at line 116, the test name at line 127, and §6.1 R7 to match.

### B2 — `ConnectionListenerRegistry` is `InheritableThreadLocal`; R2DBC events fire on Reactor scheduler threads where the registry is empty

**Plan claim (§2.3.G):**
> "The bridge listener `R2dbcConnectionLifecycleListener` resolves the active per-test `ConnectionListener` set via `for (ConnectionListener l : ConnectionListenerRegistry.INSTANCE.getConnectionListeners()) l.onConnectionCreated(connectionId);` — listeners that haven't overridden the new methods see no-ops, so JDBC paths are unaffected."

**Evidence:** `sql/sql-annotations/src/main/java/org/quickperf/sql/connection/ConnectionListenerRegistry.java:28-38, 60-67`:

```java
private static final ThreadLocal<Map<...>> CONNECTION_LISTENER_BY_TYPE_WHEN_ONE_JVM =
        new InheritableThreadLocal<...>() { ... };
...
public Collection<ConnectionListener> getConnectionListeners() {
    if (TEST_CODE_EXECUTING_IN_NEW_JVM.evaluate()) { return connectionListenersOfTestJvm; }
    return CONNECTION_LISTENER_BY_TYPE_WHEN_ONE_JVM.get().values();
}
```

`InheritableThreadLocal` propagates the parent's value **only at child thread construction time**. Reactor's `Schedulers.parallel()` and `Schedulers.boundedElastic()` worker threads are constructed lazily by Reactor's internal `ThreadFactory` — typically the very first time a Reactor publisher hits that scheduler in the JVM, well before the test thread registers any listener. Once a parallel-N or boundedElastic-M thread exists, it never re-inherits from another thread.

The consequence: when the user does `connectionFactory.create()` followed by `.close()` and the proxy's `MethodInterceptor` invokes the bridge listener, the bridge runs on a parallel scheduler thread whose InheritableThreadLocal value is `new HashMap<>()` (the `initialValue`). `getConnectionListeners()` returns an empty collection. `ConnectionLeakListener.onConnectionOpened` is **never called**, the leak counter stays at 0, and the test passes regardless of whether a connection actually leaked.

`@Execution(SAME_THREAD)` (the plan's §3.4 fallback) does not change this: it only prevents two test methods from running concurrently on different test threads. It has no effect on which thread Reactor schedules its work on.

**Why it matters:** PR-4 (`@ExpectNoConnectionLeak`) and PR-5 (`@ProfileConnection`) both rely on this dispatch path. Both will silently emit zero events on R2DBC at runtime. The IT tests under §5.2 will fail, or worse, pass for the wrong reason (no events ⇒ no leak detected ⇒ assertion of "no leak" passes).

**Fix (one of):**
- **(Preferred)** Mirror `SqlRecorderHook`'s JVM-global `CopyOnWriteArraySet<ConnectionListener>` for the *new* connection-lifecycle events. Keep `ConnectionListenerRegistry` for the existing JDBC path. The bridge `R2dbcConnectionLifecycleListener` reads from the JVM-global set, not from `ConnectionListenerRegistry`. This costs one extra registry but cleanly mirrors the existing JDBC ↔ JVM-global split that's already used for `SqlRecorder`.
- **(Alternative)** Promote the existing `ConnectionListenerRegistry` to a JVM-global `CopyOnWriteArraySet`. Risky: would change semantics for the existing `ConnectionEventsProfiler` and `ConnectionLeakListener` JDBC paths under parallel test runs. Out of scope for v2.
- **(Wrong fix)** Reactor Context is deferred to v3 — but it must not be a v2 fix unless the plan delivers it now, because the test infrastructure cannot prove R2DBC parity without it.

The plan must call this out in §2.3 explicitly and pick a strategy. Until then, every "R2DBC parity" claim in §3.6 and §6 is unverifiable.

### B3 — `JdkProxyFactory` and `JdkProxyFactoryFactory` are package-private final; a custom `ProxyFactoryFactory` cannot reuse the default and must reimplement 7 methods

**Plan claim (§2.1.A, lines 79-83):**
> "A custom `ProxyConfig.builder().proxyFactoryFactory(custom)` — installs a `ProxyFactoryFactory` whose `createResultProxy` step wraps every `Result` produced by `Statement.execute()` before it leaves the proxy."

**Evidence:**
- `target/r2dbc-proxy-sources/io/r2dbc/proxy/callback/JdkProxyFactory.java:41`: `final class JdkProxyFactory` (package-private).
- `target/r2dbc-proxy-sources/io/r2dbc/proxy/callback/JdkProxyFactoryFactory.java:27`: `final class JdkProxyFactoryFactory implements ProxyFactoryFactory`.
- `target/r2dbc-proxy-sources/io/r2dbc/proxy/callback/ProxyFactory.java:34-115`: 7 abstract methods — `wrapConnectionFactory`, `wrapConnection`, `wrapBatch`, `wrapStatement`, `wrapResult`, `wrapRow`, `wrapRowSegment`.
- `ProxyConfig.java:38, 50`: the default `ProxyFactoryFactory` is `new JdkProxyFactoryFactory()` and is consulted only if the user does not call `setProxyFactoryFactory`.

When you set a custom `ProxyFactoryFactory`, **there is no fallback to the default**. Your factory is the sole source of `ProxyFactory` instances. Your `ProxyFactory` must implement all 7 methods correctly, including the non-trivial state machine in `JdkProxyFactory.wrapConnection` (which sets up the `CallbackHandlerSupport` proxy, manages `ConnectionInfo`, and chains `ResultCallbackHandler` for sub-statements).

**Why it matters:** "Just substitute `wrapResult`" is not a one-line change. To preserve all the existing r2dbc-proxy behaviour for everything that isn't `Result`, the new `ProxyFactory` either:
1. Copies the entire `JdkProxyFactory` implementation (~200 LOC, including the package-private `CallbackHandlerSupport` invocation handler chain — also package-private so we can't call it from outside r2dbc-proxy).
2. Is implemented in `io.r2dbc.proxy.callback` package (placing our class in someone else's package; brittle but possible).
3. Uses reflection to instantiate `JdkProxyFactory` (fragile, breaks at the next minor version).
4. Avoids the custom `ProxyFactoryFactory` entirely and uses a different seam — for example, `ResultRowConverter` plus a sentinel marker per-execution to count distinct columns at row-read time. Lossy on edge cases (DDL with `Result.map`, queries that materialize without iterating rows) but does not require a new factory.

**Fix:** The plan must either (a) describe option 2 explicitly (with the package name `io.r2dbc.proxy.callback` and the `Wrapped` invariants documented), or (b) walk back to a row-counting seam in `ResultRowConverter` (option 4) and document the precision loss. As written, "install a `ProxyFactoryFactory` whose `createResultProxy` step…" is not an executable instruction.

A side note: the method on `ProxyFactory` is **`wrapResult(Result, QueryExecutionInfo, QueriesExecutionContext)`** at `ProxyFactory.java:92`. There is no method named `createResultProxy`; the plan has invented this name. Fix the API reference at §2.1 line 79 too (this is also called out in M1 below).

### M1 — `ProxyFactory` method is `wrapResult`, not `createResultProxy`

**Plan claim (§2.1.A, line 79):**
> "A custom `ProxyConfig.builder().proxyFactoryFactory(custom)` — installs a `ProxyFactoryFactory` whose `createResultProxy` step wraps every `Result`..."

**Evidence:** `ProxyFactory.java:92`: `Result wrapResult(Result result, QueryExecutionInfo executionInfo, QueriesExecutionContext queriesExecutionContext);`

There is no `createResultProxy` anywhere in r2dbc-proxy 1.1.4 (`grep -r createResultProxy target/r2dbc-proxy-sources` returns nothing).

**Fix:** Replace every "`createResultProxy`" in §2.1 / §6.1 / §7 with "`wrapResult`".

### M2 — `Result.getRowMetadata()` does not exist

**Plan claim (§2.1, line 69):**
> "R2DBC exposes column metadata only through `Result.map(BiFunction<Row, RowMetadata, T>)` or `Result.getRowMetadata()`."

**Evidence:** `target/r2dbc-spi-sources/io/r2dbc/spi/Result.java` 1.0.0 surface (verified by `grep`):
- `map(BiFunction<Row, RowMetadata, ? extends T>)` at line 56
- default `map(Function<? super Readable, ? extends T>)` at line 71
- `getRowsUpdated()`
- `filter(Predicate<Segment>)` at line 88
- `flatMap(Function<Segment, Publisher<? extends T>>)` at line 104
- No `getRowMetadata()` anywhere.

`RowMetadata` is only reachable through the `BiFunction` form of `map`. The default `Function` form receives a `Readable` (which exposes per-column accessors but not column count). `flatMap` exposes a `Result.Segment` whose `RowSegment` exposes `Row` — but again, no metadata until `Row.getMetadata()` is called per row.

**Why it matters:** This affects the §2.1.D edge cases. "Subscriber that consumes only `getRowsUpdated()` — column count is 0" is correct, but the plan's §2.1.A capture mechanism only fires inside `Result.map(BiFunction)`. If the user calls `Result.map(Function)` (the default `map`) — the path used by `databaseClient.fetch().all()` in plain Spring R2DBC code — our `BiFunction` wrapper is never invoked. Column count remains 0 even on simple `SELECT a, b, c`.

The plan says (§2.1.D) "DDL → 0; getRowsUpdated → 0" but does not list "the entire default `map(Function)` and `flatMap` paths" as zero-capture. **Most production code uses `map(Function)` via `DatabaseClient.fetch().all()`.** This is a major capability gap, not a minor edge case.

**Fix:** Either:
- Have the `Result` decorator instrument all four of `map(BiFunction)`, `map(Function)`, `flatMap`, and `filter`, intercepting `Row` access regardless of which API the user called; OR
- Document that `@ExpectMaxSelectedColumn` / `@ExpectSelectedColumn` only apply to R2DBC code paths that go through `Result.map(BiFunction)`, and recommend users use the BiFunction overload when the annotation is in play. (Less invasive but a big footnote.)

Pick one and document it in §2.1.D and `r2dbc.md`.

### M3 — `ConnectionPool extends AbstractConnectionFactory` is wrong; the class hierarchy is different

**Plan claim (§2.7, line 409):**
> "If the bean is `ConnectionPool extends AbstractConnectionFactory`, downstream consumers using `if (cf instanceof ConnectionPool)` silently miss the wrapped pool."

**Evidence:** `target/r2dbc-pool-sources/io/r2dbc/pool/ConnectionPool.java:63`:

```java
public class ConnectionPool implements ConnectionFactory, Disposable, Closeable, Wrapped<ConnectionFactory>
```

There is no class named `AbstractConnectionFactory` anywhere in r2dbc-spi 1.0.0 or r2dbc-pool 1.0.1. The pool extends `Object` directly and *implements* the four interfaces above.

This does not break the design — the CGLIB-subclass strategy still works because `ConnectionPool` is non-final — but the false hierarchy causes two downstream errors:

1. The `Wrapped<ConnectionFactory>` interface is forgotten in the plan's `addInterface(...)` list. Spring code that does `pool.unwrap()` (or anything that checks `instanceof Wrapped`) will silently miss the underlying factory through the CGLIB subclass unless the subclass implements `Wrapped` too. This is needed for `r2dbc-pool` integration with Spring Boot's auto-config tests (which call `unwrap()` to get the metrics).
2. The plan's §2.7 "addInterface(QuickPerfR2dbcProxyMarker.class)" only covers our marker. It must also include `Disposable`, `Closeable`, and `Wrapped<ConnectionFactory>` if those are checked downstream. CGLIB normally inherits these from the superclass automatically, but `Wrapped` typically requires explicit interface declaration on the proxy when used with `setInterfaces` overrides.

**Fix:**
- Replace `extends AbstractConnectionFactory` with `implements ConnectionFactory, Disposable, Closeable, Wrapped<ConnectionFactory>` in §2.7.
- Add a `unwrap()` test in PR-6 that verifies the CGLIB subclass forwards to the wrapped pool's `unwrap()` correctly.

### M4 — `SystemProperties` only reads `System.getProperty()`; the §3.5 "Spring resolution" claim is wrong

**Plan claim (§3.5):** Implies that the four new properties (`quickperf.sql.placeholder.scanner`, `quickperf.sql.profileConnection.r2dbc.operator-debug`, `quickperf.sql.r2dbc.column-count.fallback`, etc.) "pick up application.properties/application.yml/@SpringBootTest(properties=…) per the existing Boot resolver".

**Evidence:** `core/src/main/java/org/quickperf/SystemProperties.java`. Every existing entry uses the same `evaluate()` pattern:

```java
public boolean evaluate() {
    String value = System.getProperty(name);
    return Boolean.parseBoolean(value);
}
```

(Lines 26, 42, 59, 76, 93 — all five properties on this class call `System.getProperty(name)` directly.) Spring's `Environment` is **not** consulted. `application.properties` is read by Spring Boot into the Spring `Environment`, not into `System.getProperties` (unless the user explicitly does `system-properties` in their `spring.config.import` or sets `-D` on the JVM). `@SpringBootTest(properties=…)` similarly populates the Spring `Environment`, not system properties.

**Why it matters:** Users who put `quickperf.sql.placeholder.scanner=legacy` in their `application.yml` and expect the legacy regex path to activate will get the new scanner instead. Test-suite `@SpringBootTest(properties = "quickperf...=true")` will not flip the flag. The opt-out flags become unreachable in normal Spring usage, defeating the §6 risk mitigations.

**Fix:** Either:
- Add an explicit Spring-Environment bridge: in `QuickPerfR2dbcProxyBeanAutoConfiguration` (or a new config class), copy values from `Environment` to `System.setProperty(...)` at startup, OR introduce `SystemProperties.evaluate(Environment env)` that consults both. Document the precedence (env wins over -D, or vice versa).
- Or document clearly in §3.5 and `r2dbc.md`: "The opt-out flags are read via `System.getProperty(...)` only. Set them as JVM system properties (`-Dquickperf...=...`) or programmatically before tests load. They are NOT picked up from `application.properties`/`application.yml`."

Pick one. The current §3.5 wording is misleading and will produce a stream of "I set the property and it's not working" issues.

### M6 — `@Execution(SAME_THREAD)` does not protect against the actual concurrency hazard

**Plan claim (§3.4):** "All R2DBC ITs declare `@Execution(SAME_THREAD)` to prevent two test methods from registering listeners on the same JVM-global registry concurrently."

**Evidence:**
1. `pom.xml:109-110`: `<parallel>all</parallel><threadCount>5</threadCount>`. This Surefire setting applies to JUnit 4 / TestNG providers. JUnit Platform tests need `junit.jupiter.execution.parallel.enabled=true` to be parallel.
2. `junit5/junit5-extension/src/test/resources/junit-platform.properties`: only sets `junit.jupiter.extensions.autodetection.enabled=true`. Parallel execution is NOT enabled.
3. `spring/junit5-spring-boot-3-test/src/test/resources/junit-platform.properties`: same — only autodetection.
4. The only place in the repo that programmatically enables JUnit 5 parallel is `junit5/junit5-sql-test/.../QuickPerfParallelExecutionTest.java:73-76`, where it's set per-test.

So `@Execution(SAME_THREAD)` is currently a **no-op** in the modules where R2DBC tests would live (`spring/junit5-spring-boot-3-r2dbc-test/`, `sql/sql-annotations-r2dbc/`).

5. More fundamentally, `@Execution(SAME_THREAD)` only prevents **cross-test-method** concurrency. It does NOT prevent the dominant R2DBC issue: **a single test method's events firing on Reactor scheduler threads**, which is the actual cause of the InheritableThreadLocal miss in B2 above.

**Why it matters:** §3.4 is "guarding a problem that doesn't exist with a guard that doesn't help with the problem that does." A reader will see "we have a fallback for the parallel-attribution problem" and trust that R2DBC tests are robust under future parallelism, when in fact they aren't.

**Fix:**
- Remove the `@Execution(SAME_THREAD)` recommendation, OR keep it but state explicitly: "This is a defense in depth in case JUnit 5 parallel is enabled in the future. It does NOT address the within-test Reactor scheduler thread issue (see B2)."
- For the within-test issue, address B2 with a JVM-global `CopyOnWriteArraySet` mechanism for connection lifecycle events.

### M7 — Per-`QueryInfo` column count carrier vs. `ConnectionInfo.getValueStore()` lifetime

**Plan claim (§2.1.B, lines 85-91):**
> "`R2dbcExecutionAdapter` reads the captured count from `ConnectionInfo.getValueStore()` (or directly from a per-execution `ConcurrentHashMap` keyed by r2dbc-proxy execution id) and writes a `long[]`..."

**Evidence:**
- `ConnectionInfo.getValueStore()` (`target/r2dbc-proxy-sources/io/r2dbc/proxy/core/ConnectionInfo.java:101`) is **per-Connection**, lifetime = lifetime of the connection.
- A single Connection issues many queries; a single `Statement.execute()` fires `beforeQuery`/N×`eachQueryResult`/`afterQuery` and then the next statement starts. The store needs to be cleared between executions on the same Connection or it will accumulate stale counts.
- `QueryExecutionInfo.getValueStore()` does NOT exist on `QueryExecutionInfo` (only on `MethodExecutionInfo` and `ConnectionInfo` per the r2dbc-proxy 1.1.4 interface).

**Why it matters:** The plan implicitly relies on a per-execution scope for the column-count carrier, but the only available stores are per-method (transient — disposed after `afterMethod`) or per-connection (long-lived — pollutes across queries). A naive `connectionInfo.getValueStore().put("R2DBC_COL_COUNTS", new long[N])` would either be overwritten by the next query or read stale on `R2dbcExecutionAdapter.adapt`.

**Fix:** Pick one of:
- Use `MethodExecutionInfo.getValueStore()` keyed inside `Statement.execute()`'s `afterMethod`. Lifetime is correct (per call). But our `Result` decorator runs *after* `afterMethod` returns (the user subscribes to the publisher later), so the value store is already disposed by then.
- Use a separate `ConcurrentHashMap<String, long[]>` keyed by `connectionId + "#" + queryExecutionInfo.uniqueId` (introduced in 1.1.x?), populated by the decorator, drained by `R2dbcExecutionAdapter.adapt` and explicitly removed.
- Document the chosen scoping in §2.1.B with the exact key and the cleanup point.

The current "or" between two carriers in §2.1.B is hand-waved — pick one, with a justification.

### M8 — `Result.flatMap` and `Result.filter` paths are listed as "captured" but the plan doesn't say how

**Plan claim (§2.1.D, line 108):** "`Result.flatMap(...)` (deprecated in r2dbc-spi 1.0 but still supported) — also captured."

**Evidence:** `Result.java:104`: `flatMap(Function<Segment, Publisher<? extends T>>)`. Note `flatMap` is **not** deprecated in r2dbc-spi 1.0.0 (the plan misremembers — the deprecated method is `Result.map(BiFunction)` is **not** deprecated either; `getRowsUpdated()` overload returning `Publisher<Integer>` was deprecated in favor of `Publisher<Long>` but that's unrelated). The `Function<Segment, Publisher<?>>` API gives the user a `Result.Segment` directly; column-count capture must instrument the segment-handling path inside `flatMap`'s `Function`. The decorator must wrap the user's `Function` and intercept `RowSegment.row().getMetadata()` calls, similar to the `BiFunction` case for `map`.

**Why it matters:** Without explicit instrumentation of the `flatMap` path, Reactive applications using `flatMap` (e.g., `Result.flatMap(segment -> segment instanceof RowSegment ? Mono.just(((RowSegment) segment).row().get(0)) : Mono.empty())`) will not capture column count. This compounds M2.

**Fix:** Section §2.1.A must enumerate every `Result` API method that touches `Row`/`RowMetadata` and state which the decorator instruments and how. The four are: `map(BiFunction)`, `map(Function<Readable,…>)` (default impl, needs override on the decorator), `flatMap(Function<Segment, Publisher<…>>)`, `filter(Predicate<Segment>)`. Add `R2dbcResultDecoratorFlatMapIT` and `R2dbcResultDecoratorFilterIT` integration tests to §5.

### M9 — `ConnectionListener` cannot be extended in `sql-annotations-r2dbc` because all callbacks take `java.sql.Connection`

**Plan claim (§2.3.B):** "Add new no-op default methods on `ConnectionListener` for `onConnectionOpened(String connectionId)` / `onConnectionClosed(String connectionId)` / `onConnectionMethodEvent(...)`."

**Evidence:** `sql/sql-annotations/src/main/java/org/quickperf/sql/connection/ConnectionListener.java` — abstract class with ~30 callbacks, all typed `java.sql.Connection`. The plan's strategy of "new methods on ConnectionListener with `String connectionId`" works in isolation, but the existing methods also need plumbing from R2DBC bridges:
- The R2DBC bridge cannot call `onConnectionOpened(java.sql.Connection)` because it has no JDBC `Connection`. Plan says the new methods are R2DBC-friendly.
- But `ConnectionLeakListener.onClose(java.sql.Connection)` *removes* an entry from `connections: List<Connection>`. After §2.3 the leak listener tracks by string id, which means the JDBC path also needs to be retrofitted to publish to the new `String connectionId` callback. The plan says (§2.3.D) "Make `ConnectionLeakListener` track by opaque connection id (string)" — this requires JDBC-side changes too: in `QuickPerfDatabaseConnection.close(...)`, fire `onConnectionClosed(connectionIdOf(connection))` in addition to (or replacing) `onClose(connection)`.

**Why it matters:** The cross-cutting JDBC-side change is buried in §2.3.D as one bullet. PR-4 will have to touch `QuickPerfDatabaseConnection`, `QuickPerfProxyDataSource`, every JDBC test that asserts on `connections.size()`, and the existing JDBC leak ITs. Plan should split this from the R2DBC-only work, or at minimum call out the touched JDBC files.

**Fix:** §2.3 needs an explicit "JDBC-side changes" subsection enumerating which JDBC files now publish the new neutral events (and which do nothing because the abstract class supplies a default). §4.2 PR-4 file inventory should list `QuickPerfDatabaseConnection.java` (under `core/` or `sql-annotations/`), not just R2DBC files.

## Minor findings (nice-to-have)

### m1 — Existing `PlaceholderRewriter` already handles PostgreSQL `::` casts

**Plan claim (§1.2 line 44, §2.6 "Current behaviour"):** "regex passes match … PostgreSQL casts (`a::int` matches `$<digits>` if preceded by `:`). Output is wrong on all of these."

**Evidence:** `sql/sql-annotations-r2dbc/src/main/java/org/quickperf/sql/r2dbc/PlaceholderRewriter.java:50`:

```java
private static final Pattern NAMED_PLACEHOLDER = Pattern.compile("(?<!:):([A-Za-z_][A-Za-z0-9_]*)");
```

The negative lookbehind `(?<!:)` skips `::`. And the existing test `PlaceholderRewriterTest.preserves_postgres_double_colon_cast_operator()` at line 55 explicitly verifies this. The "v1 wrong on `a::int`" claim is false.

**Fix:** Remove `a::int` from the §2.6 limitations table. Replace with: "The regex correctly handles `::` casts via negative lookbehind. The scanner matches the regex behaviour on this specific case."

### m2 — `?` is not regex-rewritten today; the v1 limitation table includes a non-existent regex

**Plan claim (§1.2 line 44):** lists `?` in the placeholders that the regex misfires on.

**Evidence:** `PlaceholderRewriter.java` has only two regex passes — `NAMED_PLACEHOLDER` (`:name`) and `POSITIONAL` (`\$\d+`). The `?` character is **not** regex-rewritten at all in the existing code; the assumption is that the underlying r2dbc-proxy SQL already uses `?` (or is converted upstream). The "misfire on `?`" claim is therefore vacuous — there's nothing to misfire because there's no `?` regex.

There IS a real issue: the existing rewriter does not capture the order of `?` placeholders for the bindings list (only `:name` and `$N` populate `orderedKeys`). A SQL like `SELECT ?` produces `orderedKeys = []` and the binding is lost in the `@DisplaySql` formatting. This is a different bug from what the plan describes.

**Fix:** Reword §1.2: "The rewriter handles only `:name` and `$N`; raw `?` placeholders are passed through, but `orderedKeys` is empty for `?`-only SQL. The new scanner emits `?` placeholders in source order alongside the named/numbered ones, fixing the `@DisplaySql` rendering of pure-`?` SQL."

**Fix:** Update the §2.5 quote.

### m4 — "Activation gate short-circuits hot path" wording is misleading

**Plan claim (§2.8 line 460):** "On hot paths without the annotation, the measurement is short-circuited."

**Evidence:** `sql/sql-annotations/src/main/java/org/quickperf/sql/config/library/QuickPerfProxyDataSource.java`:

```java
public Connection getConnection() throws SQLException {
    Connection connection = theDatasourceGetsTheConnection();
    return QuickPerfDatabaseConnection.buildFrom(connection);
}
```

The `getConnection()` method always calls through to `QuickPerfDatabaseConnection.buildFrom(...)`. It does not check whether the acquisition recorder is registered. There is no "short-circuit" — the wrapping is always done.

The actual "activation gate" is at REGISTRATION time (`TestExecutionContext.buildPerfRecordersToExecute`), not at hot-path time. The recorder is created only if the annotation is present, but once created, every call pays the wrapping cost.

**Why it matters:** Reading "short-circuited on the hot path" leads a reviewer to expect a runtime conditional check that isn't there. With low-volume tests this is fine; for `@ExpectMaxQueryExecutionTime` with thousands of queries the wrapping cost is non-zero (an extra `Connection` object allocation per acquisition).

**Fix:** Reword §2.8 to: "Recorder lifecycle is gated at registration time: the acquisition recorder is constructed only when the annotation is present on the test method. Once constructed, every `getConnection()` call wraps the returned connection unconditionally; the per-call cost is one `QuickPerfDatabaseConnection` allocation."

### m5 — `R2dbcExecutionAdapter.getResult()` is null by design; "ClassCastException swallowed by printStackTrace" is wrong

**Plan claim (§2.1, line 66):** "the helper either bails through `dbExceptionHappened` (returning 0) or throws a `ClassCastException` swallowed by `printStackTrace`."

**Evidence:**
- `sql/sql-annotations-r2dbc/src/main/java/org/quickperf/sql/r2dbc/R2dbcExecutionAdapter.java:39-40` Javadoc explicitly states `getResult()` is `null`.
- `SqlExecution.java:90-92`: `executeMethodOnStatement(executionInfo)` returns `false` only for `executeQuery`. For all other R2DBC paths it returns `true`, so the `if (!executeMethodOnStatement(...))` early-return at line 73 fires before the cast.
- `SqlExecution.java:86-88`: `dbExceptionHappened(executionInfo)` returns `true` for `null` result. Returns 0.

So R2DBC always falls through `dbExceptionHappened → return 0`. The CCE path never fires.

**Why it matters:** The "or throws a `ClassCastException`" wording suggests that some R2DBC users have stack traces in their logs today. They don't. The actual v1 behaviour is silent zero, which is a more honest framing.

**Fix:** Replace "or throws a `ClassCastException` swallowed by `printStackTrace`" with "always returns 0 because `getResult()` is null; the CCE path is unreachable from the R2DBC adapter."

## Closing assessment

The v2 plan is materially better than v1. The structural designs (per-`QueryInfo` `long[]` for column count, opaque connection ids, constructor-time registration, CGLIB outer + r2dbc-proxy inner) are sound. The PR slicing is reasonable.

But three concrete blockers prevent any of PR-0/PR-2/PR-4 from being implementable as written: `ResultCallbackHandler` is `final`, `ConnectionListenerRegistry` is `InheritableThreadLocal` (so R2DBC events on Reactor threads see no listeners), and `JdkProxyFactory(Factory)` is package-private final (so a custom `ProxyFactoryFactory` must implement all 7 wrap methods from scratch). All three are problems in the r2dbc-proxy / QuickPerf API surface that the plan understates or overlooks.

In addition, §2.1's column-count capture only covers `Result.map(BiFunction)` — the *least common* user path — leaving `map(Function)`, `flatMap`, and `filter` uncovered (M2, M8). And §3.5's claim that the new system properties resolve from Spring's `Environment` is wrong; today they resolve only from `System.getProperty()` (M4), defeating the opt-out flags in normal Spring usage.

Fix the three blockers, the column-count coverage gap, the SystemProperties model, the wrong package for `SqlAnnotationsConfigs`, and the misleading API names (`createResultProxy`, `getRowMetadata`, `AbstractConnectionFactory`), and the plan is executable. Without those fixes, a subagent told to implement PR-2 will get a compile error on the first edit; one told to implement PR-4 will write code that compiles and "passes" tests by silently emitting zero events.
